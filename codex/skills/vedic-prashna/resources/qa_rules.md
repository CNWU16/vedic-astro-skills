# Prashna Q&A 追问规则

> 追问沿用原盘，不换时间挑答案。标准、Tajika、KP 的计算和结论权限保持隔离；
> 跨栈 Q&A 只并列解释，不拼票。

---

## 1. 先判追问还是新问题

追问必须仍围绕原盘的同一个可观察结果，包括：

- 澄清术语、来源支持级、输入稳定性或某条证据；
- 细化利好、阻力、现实条件和已授权 timing；
- 解释 Tajika Yoga 或 KP promise／gate；
- 比较已经生成的标准、Tajika、KP 为什么一致或不同。

以下属于新问题，回 `SKILL.md Phase 0`：

- 对象、目标结果或待执行行动改变；
- 新事实使原问题的结果语义发生实质改变；
- 从可观察结果改问秘密动机、逐字心理或长期人生趋势；
- 从原事项跳到另一个独立事项。

时间经过本身不是新问题门槛；不得使用 24 小时、3 个月等固定阈值。

---

## 2. 当前盘解析

必须先取得当前盘的明确目录或由当前对话唯一确定：

- 标准／Tajika：`prashna_<yyyymmdd_HHMMSS>_<label>/`
- KP：`kp_horary_<yyyymmdd_HHMMSS>_<label>/`

若当前对话存在多张同类盘且“这个盘”不能唯一解析，先让用户指定，不扫描所有
`prashna_*` 或 `kp_horary_*` 猜测。

输入纠错不算重起。若发现原盘使用了处理时刻、错误地点或丢失秒数，只使受影响的
下游产物失效，以原始问题时刻和实际地点重建，不把纠错盘解释成第二次占问。

## 3. 四种 Q&A 路由

| 模式 | 允许读取 | 输出权限 |
|---|---|---|
| 标准追问 | 当前 `structured_prashna.md`、`prashna_judgment_*.md`、同目录标准 Q&A | 解释标准层来源账本、三档结论和禁用 timing |
| Tajika 追问 | 当前标准目录上述文件＋`tajika_overlay.md` | 解释接触过程；不得改标准三档 |
| KP 追问 | 当前 `structured_kp.md/json`、`kp_judgment_*.md`、同目录 KP Q&A | 解释 KP promise、RP、gate、timing；不得读取标准目录 |
| 跨栈比较 | 用户或当前对话明确关联的一个标准目录和一个 KP 目录 | 分栏解释三层、一致点、冲突点；不得投票或生成混合结论 |

任何模式都禁止读取：

- `user_context.md`；
- 其他未明确选中的 Prashna／KP 目录；
- core/love/career/synastry/rectifier 等其他 skill 产物；
- 提问盘生成的本命式 Dasha、SAV、完整分盘或 transit。

## 4. 标准与 Tajika 追问

### 标准追问

按判读单已显示的 `rule_id` 回答。每个判断都须回查标准白名单字段。来源支持级
A／B 只说明题型来源覆盖，不是概率，也不自动决定“成／悬／不成”。

输入敏感性须拆开：

1. 哪个字段临界；
2. 本题账本是否实际消费该字段；
3. 主结论使用的 D1 结构是否稳定。

### 后加 Tajika

用户明确要求 Tajika 时，沿用标准盘的原始秒级时刻、地点、问者星和事项星，运行：

```bash
python scripts/build_tajika_overlay.py \
  --datetime "<原盘 YYYY-MM-DD HH:MM:SS[.ffffff]>" \
  --lat <原盘 lat> --lon <原盘 lon> --tz "<原盘 IANA>" \
  --querent-lord <Lagna lord> --matter-lord <事项宫主> \
  --out-dir "<当前 prashna_* 目录>"
```

不得重跑或覆盖 `structured_prashna.md`，也不存在 `--enable-tajika`。

## 5. KP 追问与后加 KP

已有 KP 盘的追问沿用原 KP number、判盘时刻、地点和 topic，不重新选号码。
先核对原盘的 outcome scope。`love-materialization` 的结论只回答是否建立明确、
双方确认并持续推进的恋爱关系；不得外推为是否会恢复联系、互动回暖或恢复暧昧。
若原问题只是这些低门槛结果，却误用了 `love-materialization`，保留原计算文件作为
高门槛题型结果，但明确宣布它不能回答原低门槛问题；不得据此改判或重算。

标准盘之后想查看 KP，不是把 KP 字段追加到原盘，而是新建独立 KP 栈。必须：

1. 用户亲自给出 1–249；
2. 记录开始判断 KP 问题的秒级时刻和实际地点；
3. 让用户确认同一个可观察结果对应的 outcome scope；
4. 只有 outcome scope 与已支持 topic 完全一致时才起盘；
5. 写入独立 `kp_horary_*` 目录。

这不允许 KP 回写或改判标准结论。

## 6. 宽泛跨栈追问

用户问“整体怎么看”“三层怎么说”“为什么不同”时，若相关结果均已存在，按固定
结构回答：

```markdown
## 标准层
<来源支持级、三档结论、规则账本摘要>

## Tajika
<直接接触、传递/援助/阻断、timing 状态>

## KP
<独立 promise 状态、正反宫命中、gate、timing 状态>

## 一致与冲突
- 一致点：<逐项列出>
- 冲突点：<逐项列出>
- 边界：三层不投票；标准层仍是标准结论，KP 保持独立。
```

跨栈比较只能在 Q&A 协调层读取明确关联的文件。默认只在聊天显示；用户要求保存时，
写入其明确授权的共同私人父目录，文件名 `qa_compare_<label>.md`，不得写进任一栈
目录造成后续偷读。

---

## 7. Timing 追问

- 标准层未启用生产级 timing：直接说明不可用，不重起盘补日期；
- Tajika 仅在主星直接 Itthasala 时显示 `度差 × 12 日` 原典比例候选；
- KP 只在正向 promise、逆行门清空、node 路径完整和 RP／period／transit 条件满足
  时显示候选；
- 一个栈没有 timing 时，不得借另一个栈的日期冒充本栈答案。

---

## 8. Q&A 文件

单栈追问写入当前栈目录的 `qa_<主题短标签>.md`：

```markdown
# Q&A：<主题>

**关联盘**：<精确目录名>
**追问时刻**：<YYYY-MM-DD HH:MM:SS>
**追问类型**：[澄清／细化／拓展]
**目标栈**：[标准／Tajika／KP]

## 问
<用户原问>

## 答
<白话解释>

## 引用
- <文件> §<段>：<字段或结论>
```

多轮同主题可追加；不同主题拆文件。Q&A 不改写原结构盘或原判读单。
